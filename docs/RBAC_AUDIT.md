# تدقيق RBAC مؤسسي شامل — منصة «منارة»

> **النوع:** تدقيق صلاحيات كامل عبر كل الطبقات (Frontend/Backend/API/Database/Business Logic) + إصلاح فعلي + اختبارات محاولة تجاوز.
> **التاريخ:** 2026-07-04 · **المنهجية:** جرد كل نقطة نهاية API × كل الأدوار التسعة، ثم التحقق من التطبيق الفعلي لـ ownership/tenant في طبقة الخدمة (لا الاكتفاء بديكوريتر `@Roles`) — الافتراض الأساسي: **النظام غير آمن حتى يثبت العكس**.

---

## 1. الملخص التنفيذي

فحص شامل لكل الأدوار التسعة عبر 15 وحدة API كشف **4 ثغرات Critical** و**3 ثغرات High** — جميعها من نمط IDOR/Broken Access Control الكلاسيكي: التحقق من الدور (`@Roles`) كان صحيحًا في كل مكان، لكن التحقق من **ملكية البيانات** (ownership) داخل طبقة الخدمة كان **ناقصًا لدور STUDENT بالكامل** بسبب غياب رابط بنيوي بين حساب الدخول وسجل الطالب الفعلي، إضافة لثغرات تسريب مالي للمعلم عبر لوحة المعلومات والبحث.

**كل الثغرات أُصلحت وأُثبتت بإصلاحها عبر 11 اختبار تلقائي جديد** (محاولات تجاوز فعلية: IDOR، تصعيد صلاحيات، تخمين معرّفات، حقن معرّفات عبر المؤسسات) — إجمالي اختبارات الـ API الآن **26/26 خضراء**.

| الخطورة | العدد | الحالة |
|---|---|---|
| 🔴 Critical | 4 | ✅ كلها مُصلَحة |
| 🟠 High | 3 | ✅ كلها مُصلَحة |
| 🟡 Medium | 2 (تحسينات نطاق) | ✅ كلها مُصلَحة |

## 2. السبب الجذري

```prisma
model Student {
  guardianUserId String?  // ربط بحساب PARENT — موجود ✅
  studentUserId  String?  // ربط بحساب STUDENT — لم يكن موجودًا إطلاقًا ❌
}
```

لم يوجد أي حقل يربط حساب دخول بدور `STUDENT` بسجل الطالب الفعلي الخاص به. نتيجة ذلك: كل نقطة نهاية تتحقق من ملكية PARENT (`guardianUserId === user.id`) كانت **تتجاهل STUDENT تمامًا** أو تمرّره بشرط `true` (بلا أي قيد) — لأن الكود لم يكن يملك أصلًا وسيلة لمعرفة "هل هذا سجل الطالب الحالي؟".

**الإصلاح الجذري:** إضافة `Student.studentUserId` (فريد 1:1) + دالتا مساعدتان مركزيتان في `common/types.ts`:
```ts
ownStudentWhere(user)         // { studentUserId: user.id } لـ STUDENT، { guardianUserId } لـ PARENT
ownStudentRelationWhere(user) // نفس الشيء عبر علاقة student متداخلة (حضور/رسوم)
```
كل نقطة نهاية كانت ناقصة استخدمت هاتين الدالتين بدل شروط مكتوبة يدويًا متفرقة — يمنع تكرار الخطأ مستقبلًا.

## 3. الثغرات المُكتشفة والمُصلَحة (بالتفصيل)

### 🔴 CRITICAL

**C1 — الطالب يرى ملف أي طالب آخر في مؤسسته بالكامل**
`GET /students/:id` — `students.service.ts` كان يتحقق من ملكية PARENT فقط؛ STUDENT يمرّ بلا أي قيد. **الأثر:** طالب يكتب أي `id` (بتخمين تسلسلي `st1000`, `st1001`...) ويرى: الرصيد المالي الكامل، الملاحظات الصحية، العنوان، سجل الحضور، كل نتائج الامتحانات، الخصومات، والوثائق المرفوعة لأي زميل.
**الإصلاح:** `if (user.role === "STUDENT" && student.studentUserId !== user.id) throw ForbiddenException`.
**الاختبار:** #16 — طالب يرى ملفه (200)، يُمنع من ملف زميله (403).

**C2 — كشف نتائج الامتحان يعرض للطالب الشعبة كاملة**
`GET /exams/:id/results` — منطق حجب الديون كان: `user.role === "PARENT" ? ownership : true`. الشرط `true` يعني **بلا أي فلترة** لدور STUDENT. **الأثر:** طالب يفتح كشف أي امتحان ويرى أسماء ودرجات **كل** زملائه في الشعبة.
**الإصلاح:** الشرط أصبح `r.student.studentUserId === user.id`، مع الحفاظ على صحة الترتيب (rank) المحسوب على الشعبة كاملة *قبل* التصفية (خلاف ذلك كان سيظهر ترتيب خاطئ دائمًا "1 من 1").
**الاختبار:** #17 — الطالب يرى صفًا واحدًا فقط، بترتيب صحيح (2 من 2) رغم رؤيته سجله وحده.

**C3 — تنزيل كشف درجات أي طالب عبر تخمين المعرّف**
`GET /exams/:id/results/:studentId/card` — نتيجة مباشرة لـ C2: بما أن `results()` كانت تعيد الشعبة كاملة لـ STUDENT، فإن تمرير `studentId` لزميل كان ينجح ويُنتج PDF كشف درجاته.
**الإصلاح:** تلقائي بعد إصلاح C2 (الدالة تبحث الآن ضمن نتائج مُقيَّدة أصلًا) + تحقق إضافي من تبعية الطالب للمؤسسة.
**الاختبار:** #18 — طلب كشف زميل → 404؛ طلب كشف نفسه → 200.

**C4 — لوحة المعلومات تسرّب الإيرادات والبيانات الإدارية لأربعة أدوار**
`GET /dashboard` — بلا أي `@Roles` (أي دور مصادَق يمر)، وفرع "أدوار المؤسسة" الواحد كان يخدم `SCHOOL_ADMIN/ACCOUNTANT/TEACHER/STUDENT/HR/DRIVER` معًا بنفس الحمولة: `collected` (المحصَّل)، `outstanding` (الذمم)، `collectionRate`، `receiptsThisMonth`، `pendingAdmissions`. **الأثر:** أي طالب أو معلم أو سائق أو موظف موارد بشرية يفتح لوحة المعلومات ويرى **إجمالي إيرادات المدرسة وذممها المالية** — خرق مباشر لمتطلب "Students/Teachers must NEVER access: Revenue/Financial dashboard".
**الإصلاح:** إعادة بناء كاملة — كل دور يستلم حمولة بشكل مختلف تمامًا (لا حقل مالي مشترك يُخفى بالواجهة فقط): `platform` (SUPER_ADMIN/AUDITOR) · `school` (SCHOOL_ADMIN/ACCOUNTANT فقط، بلا تغيير) · `teacher` (شعبه فقط) · `student` (سجله فقط) · `hr` (عدد الموظفين فقط) · `driver` (مساراته فقط) · `wards` (PARENT، بلا تغيير). أُضيف `@Roles` صريح بالأدوار التسعة (fail-closed: أي دور مستقبلي يُحجب تلقائيًا حتى يُقرَّر شكل حمولته).
**الاختبار:** #19 — يتحقق أن حمولتي الطالب والمعلم لا تحويان `collected|outstanding|collectionRate|pendingAdmissions` إطلاقًا (فحص نصي شامل على كامل الاستجابة، لا حقلًا بعينه).

### 🟠 HIGH

**H1 — إرسال رسائل لمستخدمين من مؤسسة أخرى**
`messages.service.ts` (جمهور `INDIVIDUALS`) كان يستخدم `userIds` من العميل مباشرة بلا أي تحقق من `tenantId`. **الأثر:** أي دور يملك صلاحية إرسال رسائل (SCHOOL_ADMIN/ACCOUNTANT/TEACHER) يستطيع دفع إشعارات (وتفعيل قنوات خارجية SMS/WhatsApp لاحقًا) لأي مستخدم في **أي مؤسسة أخرى بالنظام**، بما فيها حسابات AUDITOR/SUPER_ADMIN.
**الإصلاح:** تصفية `userIds` عبر `User.findMany({ id: { in }, tenantId })` ورفض الطلب (400) إن نقص عدد المتحقَّق منهم عن المرسَل.
**الاختبار:** #20.

**H2 — بحث المعلم يكشف سندات القبض والمبالغ المالية**
`search.module.ts` — نتائج البحث العام شملت `payments` (رقم السند + المبلغ) لكل الأدوار المسموحة بالبحث، ومنها TEACHER الذي لا يحق له الاطلاع على أي بيانات مالية إطلاقًا.
**الإصلاح:** استبعاد `payments` كليًا لدور TEACHER، وتقييد نتائج `students` بشعبه فقط.
**الاختبار:** #21.

**H3 — المعلم يستطيع عرض تقرير حضور أي شعبة في المؤسسة**
`attendance.service.ts` دالة `report()` — كانت الدالتان `sheet()`/`saveBulk()` تتحققان من ملكية الشعبة للمعلم، لكن `report()` (تقارير الحضور بفلاتر) لم تتحقق إطلاقًا؛ معلم يمرّر `sectionId` لشعبة زميل ويرى حضور طلابها.
**الإصلاح:** نفس فحص `section.teacherId === user.id`؛ بلا `sectionId` محدد يُقيَّد تلقائيًا بشعبه فقط.
**الاختبار:** #22.

### 🟡 تحسينات نطاق (إغلاق فجوات وظيفية موثّقة في المواصفة)

**M1 — الطالب بلا وصول لحضوره الخاص** — كان مستبعدًا كليًا من `/attendance/report` رغم أن المواصفة تنص صراحة "Students may ONLY access: ... Their own attendance". أُضيف STUDENT مع تقييد `ownStudentRelationWhere`. **الاختبار:** #23.

**فحص إضافي — المعلم يرى امتحانات/درجات شعب لا يملكها (قراءة)** — `exams.service.ts list()`/`results()` لم تُقيَّدا لشعبة المعلم (فقط الكتابة `create()`/`upsertResults()` كانت مُقيَّدة من تصليب سابق). أُضيف نفس القيد للقراءة.

**تحسين — وصول الطالب لأقساطه/سنداته الخاصة** — لم يكن للطالب أي مسار لرؤية "his own payments/invoices" رغم أنه مطلوب صراحة بالمواصفة؛ وُسِّع نمط PARENT الموجود (`ownStudentRelationWhere`) ليشمل STUDENT في `listRecords`/`listPayments`/`receiptData`. **الاختبار:** #24.

## 5. جولة تجاوز ثانية (بعد الإصلاح) — تُنفَّذ فعليًا في نفس السويت

طُبِّقت جولة تجاوز منهجية بعد كل إصلاح (اختبار #25) تُطلق 15 طلبًا بحساب STUDENT على مسارات إدارية/مالية حساسة (`/tenants`, `/audit`, `/employees`, `/students` [قائمة], `POST /students`, `DELETE /students/:id`, `/payments`, `/payments/:id/void`, `/discounts`, `PUT /exams/:id/results`, `/attendance/bulk`, `/ai/generate`, `/tenants/mine/settings`, `/routes`) وتتحقق أن **كل واحد منها** يرفض بـ 401/403 — صفر استثناءات. إضافة اختبار تخمين معرّفات عبر المؤسسات (#26): مدير مؤسسة أ يُمنع من أي كيان في مؤسسة ب (404/403).

**النتيجة: لا توجد ثغرة معروفة متبقية بعد هذه الجولة.**

## 6. ما تم فحصه ولم يُظهر ثغرة (لتوثيق نطاق التغطية)

- عزل المستأجرين الأساسي (`tenantWhere`) — سليم وسبق اختباره صراحة.
- ترقيم السندات، إلغاء السندات (`voidPayment` ضمن `tenantWhere` دفاعيًا)، الخصومات، القبول → تحويل لطالب — كلها scoped بشكل صحيح.
- الإشعارات (`notifications.service.ts`) — ذاتية النطاق فعليًا عبر `userId: user.id` في كل استعلام (بلا حاجة لـ `@Roles` لأن الحماية بنيوية).
- `tenants.controller.ts` — SCHOOL_ADMIN يُمنع صراحة من عرض تفاصيل مؤسسة أخرى.
- رفع/تخزين الملفات — لا يوجد أي مسار تقديم ثابت (`static serving`) لمجلد `uploads/`؛ لا نقطة تنزيل مكشوفة حاليًا (فجوة وظيفية موثقة، ليست ثغرة).

## 9. الخلاصة

النظام كان يطبّق RBAC على مستوى الدور (`@Roles`) بدقة، لكنه كان يفتقر لتحقق **الملكية** (ownership) الكامل لدور STUDENT بسبب غياب بنيوي (لا حقل ربط)، إضافة لتسريبين ماليين محدودين للمعلم. بعد هذه الجولة: **كل الأدوار التسعة تُطبَّق عليها قاعدة "أقل امتياز" فعليًا على مستوى البيانات لا الدور فقط**، مثبتة بـ 26 اختبارًا آليًا يتضمن محاولات تجاوز فعلية (IDOR، تصعيد صلاحيات، تخمين معرّفات، حقن معرّفات عبر المؤسسات) — لا توصية معلّقة، ولا ثغرة معروفة متبقية.
